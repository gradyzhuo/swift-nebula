# Comet — Async Messaging (MQ + Pub/Sub)

**Date:** 2026-03-30
**Status:** Draft

## Overview

Add at-least-once async messaging to Nebula via a new `Comet` client type and a new `BrokerAmas` server-side component. The same infrastructure supports both point-to-point MQ (one consumer processes the message) and pub/sub (all subscriber groups receive the message), unified under a topic + subscription model.

`LoadBalanceAmas` is not modified — it remains a pure load balancer.

## Background

`RoguePlanet` is optimised for request-reply. Async workloads (background jobs, event-driven processing, event notifications) need a different model. Persistence is out of scope for Layer 1 — exposed as an optional `QueueStorage` plugin.

## Design

### Unified Topic + Subscription Model

```
Topic（namespace）
└── Subscription A  → Stellar 1, Stellar 2  (competing consumers = MQ)
└── Subscription B  → Planet X              (independent consumer = Pub/Sub)
```

- One message fans out to **all** Subscriptions
- Within each Subscription, one consumer is picked (round-robin)
- One Subscription → MQ behaviour; multiple Subscriptions → Pub/Sub behaviour

### Two Amas Types

```
Amas (protocol)
├── LoadBalanceAmas   — existing, load balancer only, unchanged
└── BrokerAmas        — new, handles queue / topic / dispatch / ACK / park
```

Galaxy creates the appropriate Amas based on the connection scheme (see URL section).

### URL Scheme Extension

```
nmtp://ingress:2240/production/ml/embedding
  → RoguePlanet RPC → LoadBalanceAmas

nmtp+broker://ingress:2240/production/orders
  → Comet MQ/Pub/Sub → BrokerAmas
```

A namespace can hold both simultaneously:
```
production.orders
├── LoadBalanceAmas   (nmtp://)          → synchronous RPC
└── BrokerAmas        (nmtp+broker://)   → async messaging
```

### Message Flow

```
Comet ──.enqueue──→ Ingress ──→ Galaxy ──→ BrokerAmas
         ←─ "queued" ──────────────────────┘  (timeout if unreachable)

BrokerAmas ──.enqueue──→ Subscriber (Stellar or Planet)
BrokerAmas ←──.ack──────── Subscriber (processing complete)
BrokerAmas removes from active storage

If retries exhausted:
BrokerAmas ──→ parked storage
```

### New Message Types

```
+ .enqueue     // Comet → BrokerAmas  AND  BrokerAmas → Subscriber
+ .ack         // Subscriber → BrokerAmas (processing complete)
+ .subscribe   // Subscriber → BrokerAmas (join a subscription group)
+ .unsubscribe // Subscriber → BrokerAmas (leave)
```

### New Types

#### `QueuedMessage`
```swift
public struct QueuedMessage: Codable, Sendable {
    public let id: UUID              // = Matter.messageID, dedup key
    public let namespace: String
    public let method: String
    public let arguments: [EncodedArgument]
    public let enqueuedAt: Date
    public var retryCount: Int
}
```

#### `RetryPolicy`
```swift
public struct RetryPolicy: Sendable {
    public let maxRetries: Int       // default: 3
    public let ackTimeout: Duration  // default: 30s

    public static let `default` = RetryPolicy(maxRetries: 3, ackTimeout: .seconds(30))
}
```

#### `QueueStorage` (plugin protocol)
```swift
public protocol QueueStorage: Sendable {
    func append(_ message: QueuedMessage) async throws
    func remove(id: UUID) async throws
    func pendingMessages() async throws -> [QueuedMessage]
}
```

`BrokerAmas` holds two `QueueStorage` instances — `active` and `parked` — independently. Allows mixing backends (e.g. in-memory active + SQLite parked).

Default: `InMemoryQueueStorage`. Optional externals: SQLite, file, Redis (separate packages).

#### `BrokerAmas` actor
```swift
public actor BrokerAmas: Amas {
    let active: any QueueStorage
    let parked: any QueueStorage
    let retryPolicy: RetryPolicy
    // internal: subscription registry, dispatch loop
}
```

#### `Comet` actor
```swift
public actor Comet: Astral {
    public func enqueue(
        namespace: String,
        method: String,
        arguments: [Argument] = []
    ) async throws  // confirms "queued", timeout if BrokerAmas unreachable
}
```

No direct Stellar connection. No per-namespace connection cache.

**Stellar as publisher**: Stellar creates a `Comet` instance internally to emit events. No new publishing capability needed on `ServiceStellar`.

#### `Satellite` actor
```swift
public actor Satellite: Astral {
    // Connects to Ingress, subscribes to a topic, receives server-push events.
    public func events(
        topic: String,
        subscription: String
    ) -> AsyncStream<Matter>
}
```

`Satellite` maintains a persistent connection and receives unsolicited `.event` messages pushed by `BrokerAmas`. `AstralCategory` gains a `.satellite` case.

**NMTClient extension required**: `NMTClientInboundHandler` gains a second path — incoming `Matter` with no matching pending request is forwarded to an event stream (server push). Current reply-matching path is unchanged.

#### `Observatory` actor
```swift
public actor Observatory {
    public func parkedMessages(namespace: String) async throws -> [QueuedMessage]
    public func replay(namespace: String) async throws
    public func discard(id: UUID, namespace: String) async throws
}
```

### Changes to Existing Types

**`StandardGalaxy`**
- Add `retryPolicy: RetryPolicy` global default + per-namespace override
- `configure(namespace:retryPolicy:)` for override
- On `.enqueue` → route to `BrokerAmas` (created on demand for `nmtp+broker://` namespaces)
- On `.register` → route to `LoadBalanceAmas` as before (unchanged)

**`ServiceStellar`**
- Handle `.enqueue` in `coreDispatch` → dispatch to service/method → reply `.ack`
- Middleware chain applies to `.enqueue` the same as `.call`

**`NebulaURI`**
- Parse `nmtp+broker://` scheme in addition to `nmtp://`
- Path semantics are scheme-dependent:
  - `nmtp://host:port/galaxy/namespace` → `namespace`
  - `nmtp+broker://host:port/galaxy/topic/subscription` → `topic` + `subscription`

**`NMTClient+Astral`**
- Add `enqueue(namespace:method:arguments:)` on `NMTClient<IngressTarget>`
- Add `subscribe(topic:subscription:)` on `NMTClient<IngressTarget>`

**`NMTClientInboundHandler`**
- Add second path: `Matter` with no matching pending request → forward to event stream (server push)
- Existing reply-matching path unchanged

## Decisions

1. **`Comet` is a concrete actor implementing `Astral` directly** — no protocol, not related to `Planet`. `AstralCategory` gains a `.comet` case.

2. **`LoadBalanceAmas` is not modified** — MQ/Pub/Sub logic lives in the new `BrokerAmas`. Single responsibility.

3. **`BrokerAmas` is a new dedicated Amas type** — manages topic + subscription model, dispatch loop, ACK, retry, park.

4. **`RetryPolicy` cascades: Galaxy global default + per-namespace override.**

5. **Active and parked are separate `QueueStorage` instances** held by `BrokerAmas`.

6. **URL scheme differentiates connection type**: `nmtp://` → RPC/`LoadBalanceAmas`; `nmtp+broker://` → MQ+Pub/Sub/`BrokerAmas`. Same namespace can have both.

7. **`Observatory` manages parked messages** — `Comet` = producer only; `Observatory` = operator.

8. **Subscription creation is implicit** — consumer joins a named group; `BrokerAmas` creates it on first join.

## Decisions (continued)

5. **Stellar publishes via Comet** — Stellar creates a `Comet` instance internally. No new publish capability on `ServiceStellar`.

6. **`Satellite` is the subscriber role** — maintains a persistent connection, receives server-push `.event` messages from `BrokerAmas` via `AsyncStream`. `AstralCategory` gains `.satellite` case.

7. **URL scheme determines path semantics**:
   - `nmtp://` → `/galaxy/namespace`
   - `nmtp+broker://` → `/galaxy/topic/subscription`

8. **Subscription creation is implicit** — `BrokerAmas` creates the group on first `.subscribe`.
